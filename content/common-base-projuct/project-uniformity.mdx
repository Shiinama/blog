# 有关项目的整体性的问题

实际上，整个最简AI项目是需要靠着能够批量化、快速替换、快速生成的方向去靠近的。那么因此在编写项目的过程中，有一些共识性的理解和偏向AI去生成的细节。

那么这一章主要是给大家补充一些我对于快速建站的一些看法，以及对于整个项目在最终成型的形态描述。

> 此章不进入正式章节，只是一个思考科普文

## AI建站？

实际上，现在市面上有各种各样的AI建站工具以及AI编辑，包括我自己也用，我甚至还有v0和cursor的会员。

但在项目过于复杂的情况下，他们在整体生成的表现就会略差，这是由模型上下文的长度所决定的短板，还有一个很痛苦的问题就是模型本身的数据不够新，以及在多层复杂难度的表现很难过让我。

在这些问题的叠加下，就变成了不能完全用工具建站，但没关系这个模版项目就是要解决这些短板问题，用比建站工具更快的时间去完成一个闭环的商业项目。

因此要达到这种效果，就需要有一些共识，[本章tag](https://github.com/Shiinama/next-cloudflare-template/tree/v3.0.0)

## 样式的共识

显然，我们不可能说我出10个项目，就需要写10个样式的模版，那最好的方式就是完全的使用`主题配色`，下面的css中是项目的主要配色，我们在后续项目的时候，这可以直接让AI生成一套配色，或者去[shadcn主题10选1](https://ui.shadcn.com/docs/theming)。

```css
    --background: 240 10% 3.9%;
    --foreground: 0 0% 98%;
    --card: 240 10% 3.9%;
    --card-foreground: 0 0% 98%;
    --popover: 240 10% 3.9%;
    --popover-foreground: 0 0% 98%;
    --primary: 0 0% 98%;
    --primary-foreground: 240 5.9% 10%;
    --secondary: 240 3.7% 15.9%;
    --secondary-foreground: 0 0% 98%;
    --muted: 240 3.7% 15.9%;

    --accent: 240 3.7% 15.9%;
    --accent-foreground: 0 0% 98%;
    --destructive: 0 62.8% 30.6%;
    --destructive-foreground: 0 85.7% 97.3%;
    --border: 240 3.7% 15.9%;
    --input: 240 3.7% 15.9%;
    --ring: 240 4.9% 83.9%;
    --chart-1: 220 70% 50%;
    --chart-2: 160 60% 45%;
    --chart-3: 30 80% 55%;
    --chart-4: 280 65% 60%;

    --sidebar-accent: 240 3.7% 15.9%;
    --sidebar-accent-foreground: 240 4.8% 95.9%;
    --sidebar-border: 240 3.7% 15.9%;
    --sidebar-ring: 240 4.9% 83.9%;
  }

```

同学们可能会问，为什么他们提供光暗双配色，但项目里只有一种？因为双配色不是必要的，会非常浪费时间去细调样式，因为要考虑两套配色（光暗统一性），同时也增加了AI生成的Token成本，所以项目只有一种配色。

正好也解释一下这些颜色变量的意思，方便同学们后面使用以及扔给AI：

- `--background`/`--foreground`：页面基础背景和文本颜色
- `--card`/`--card-foreground`：卡片组件的背景和文本颜色
- `--popover`/`--popover-foreground`：弹出组件的背景和文本颜色
- `--primary`/`--primary-foreground`：主要按钮/元素的背景和文本颜色
- `--secondary`/`--secondary-foreground`：次要按钮/元素的背景和文本颜色
- `--muted`/`--muted-foreground`：弱化元素的背景和文本颜色
- `--accent`/`--accent-foreground`：强调元素的背景和文本颜色
- `--destructive`/`--destructive-foreground`：危险操作的背景和文本颜色
- `--border`：边框颜色
- `--input`：输入框边框颜色
- `--ring`：聚焦环颜色

## 文案类的共识

那么在样式背后就是文案了，这跟I18n也息息相关，因为I18N这种JSON结构的数据类型，它是集中的，不需要到每个具体页面去用AI生成文案，更利于统一生成不同文案的，在我们完成整个项目后，我们会发现只有少部分的文案是需要根据不同业务和功能生成的（而CLI也是为此而生的）。

因此我会在未来的编写模版的时候，保证文案的通用性，其余不通用的部分采用AI去生成，在用CLI去跑。

## AI模型的共识

显然，在前面的部分我都是采用的`cloudflare`的AI模型功能去生成的，因为它免费，又好集成。

但并不是说`cloudflare`的模型就有多好，只能说用得非常的勉强+效果极差。

在面向用户的功能的时候尽可能还是需要挑选合适的模型，对于文字模型可以去我写的[平台类demo站](https://xibaoyu.xyz/)挑选和使用。

并不是说`Open AI`系模型就是最好的，有很多模型能在只有10几分支1的价格就能达成GPT4的效果，并不是无脑选模型。

图像系模型百花齐放，要更高的定制化需要了解`ComfyUI`。

声音模型主要分为：文本转语音(TTS)模型（适用于内容朗读、虚拟助手和有声内容制作）、语音克隆模型（基于少量样本复制特定声音，适用于个性化内容创作、配音替换和品牌声音定制）、实时语音合成（数字人）。，也是需要用什么就玩什么。

视频模型~噱头，工程商业不成熟至少现在是。

> 其实讲AI就能讲的太多了，这里讲的都是简单应用层的东西，最好还是用到什么讲什么~

## AI生成内容的共识

任何一个站需要起来，都是需要内容的，对于Google来说，肯定一个站全是精品内容会获得高排名。

但显然不可能的，时间和精力成本不允许，要不就是AI写+人修，要不就是全AI写内容。

Google年年都在说打击AI，但显然打击的是垃圾内容，不是打击AI，即使人写了大量垃圾也一样。

打击AI内容，那市面上，这些用AI生成的文章全都可以去世了。而对于最快、最简单，免费起量的方式，只能AI写内容。

> 我自己的项目每2天找1000个关键词，1000篇文章~。

## 最终成型的项目预期

A: 广告流量站，不需要Token、付费、订阅、各种条款这几个模块。仅有核心功能+登录+限制次数+大量内容。核心功能看对于场景和技术的熟悉程度，大概半个小时能完成。
B: 付费订阅完整商业站，也就是模块全要，但因为模块都是固定的，大概加起来20分钟搞定。核心功能半个小时，总时间50分钟一个项目。

两者的每日运营时间都相似，找关键词一天50个+生成10分钟。各种渠道找+发布50分钟（Twitter、Reddit等发起来）。

无论如何每个项目只要长期运营都等于是正资产和低投入的，一个域名70块，cloudflare生态全免费，想用更好的模型也有超便宜的AI API（现在这时代真是资源太充沛了）。

如果做起来有收益，还可以投流，和找KOL/KOC。

再加上整个社群大量的项目集群效应，外链、社媒推荐、子频道、外置模块等，会起到一个放大效果，项目的成功率会随着越来越多的项目逐渐上涨，在我看来这是一件必成的事情。

这个公式是：一个合理足够快的模版项目 + 无限低成本的尝试 + 积累的渠道 + 长期的运营 + AI的多样性 = 现金奶牛
